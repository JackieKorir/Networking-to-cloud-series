Section 1: VPC Fundamentals

Building Your Private Cloud Network

What is a VPC?

A Virtual Private Cloud (VPC) is your logically isolated network within the AWS cloud. Think of it as your own private data center, but without the physical hardware. A VPC gives you complete control over your virtual networking environment, including IP address ranges, subnets, route tables, and network gateways.

VPCs are the foundation of AWS networking. When you deploy any resource in AWSâ€”whether it's an EC2 instance, RDS database, or Lambda function with VPC accessâ€”it exists within a VPC. Understanding VPCs is essential for building secure, scalable cloud architectures.

ğŸ”‘ Key Definition: VPC = Your Private Network in AWS. It's logically isolated from other AWS customers with complete control over your networking configuration, including IP addressing, routing, and security.

AWS VPCs abstract away physical networking hardware, replacing it with software-defined networking that's more flexible, scalable, and cost-effective.

Default VPC vs Custom VPC

Default VPC Characteristics

Every AWS account comes with a default VPC in each region. It's pre-configured and ready to use immediately, making it perfect for getting started quickly with AWS services.

Default VPC Configuration:

  • CIDR Block: 172.31.0.0/16 (pre-assigned, cannot be changed)
  • Subnets: One public subnet in each Availability Zone
  • Internet Gateway: Already attached and configured
  • Route Table: Main route table configured for internet access
  • DHCP Options Set: Pre-configured for DNS resolution
  • Use Case: Quick testing, learning, and development environments

Custom VPC Advantages

For production workloads and enterprise applications, custom VPCs are essential. They provide the control and flexibility needed for complex architectures and security requirements.

Why Custom VPCs Matter:

  • IP Address Control: Choose your own CIDR blocks from RFC 1918 private ranges
  • Network Architecture: Design multi-tier applications with public/private subnets
  • Security: Implement network segmentation and isolation best practices
  • Compliance: Meet specific regulatory and organizational requirements
  • Scalability: Plan for future growth with proper IP space allocation
  • Multiple Environments: Separate dev, staging, and production networks

ğŸ’¡ When to Use Default vs Custom VPC

Use Default VPC for:

  • Learning AWS and experimenting with services
  • Quick proof-of-concepts and demos
  • Temporary development and testing

Use Custom VPC for:

  • All production applications
  • Enterprise workloads with security requirements
  • Multi-tier application architectures
  • Hybrid cloud connectivity (VPN, Direct Connect)

CIDR Blocks: IP Address Planning

CIDR (Classless Inter-Domain Routing) notation is how you define IP address ranges in your VPC. Proper CIDR planning is crucial because it's difficult to change later, and running out of IP addresses can limit your ability to scale.

Understanding CIDR Notation

Example: 10.0.0.0/16 Components: â”œâ”€â”€ IP Address: 10.0.0.0 (base network address) â””â”€â”€ Prefix: /16 (subnet mask, determines network size) The prefix number indicates how many bits are fixed: â€¢ Smaller prefix = MORE IP addresses (larger network) â€¢ Larger prefix = FEWER IP addresses (smaller network) Common VPC CIDR Sizes: /16 = 65,536 IP addresses (Recommended for large deployments) /20 = 4,096 IP addresses (Medium-sized environments) /24 = 256 IP addresses (Small environments) /28 = 16 IP addresses (Minimum allowed in AWS)

AWS VPC CIDR Requirements

  • Size Range: Between /16 (largest) and /28 (smallest)
  • Private Ranges: Should use RFC 1918 addresses:
    • 10.0.0.0/8 (10.0.0.0 - 10.255.255.255)
    • 172.16.0.0/12 (172.16.0.0 - 172.31.255.255)
    • 192.168.0.0/16 (192.168.0.0 - 192.168.255.255)
  • No Overlap: Cannot overlap with other VPCs if using VPC Peering or Transit Gateway
  • Secondary CIDRs: Can add additional CIDR blocks later (up to 5 total)
âœ… Best Practice: CIDR Planning Strategy

Start with a /16 CIDR block to maximize flexibility. Use a systematic approach:

  • VPC 1 (Production): 10.0.0.0/16
  • VPC 2 (Staging): 10.1.0.0/16
  • VPC 3 (Development): 10.2.0.0/16
  • VPC 4 (Management): 10.3.0.0/16

This gives you room for 256 VPCs with 65,536 IPs each, and they'll never overlap.

Tenancy: Default vs Dedicated

Instance tenancy determines whether your EC2 instances run on shared or dedicated physical hardware. This choice affects both security posture and costs.

Feature Default Tenancy Dedicated Tenancy
Hardware Shared with other AWS customers Physical servers dedicated to your account
Isolation Logical isolation via hypervisor Physical isolation
Security Highly secure (AWS-managed) Adds physical separation layer
Cost Standard EC2 pricing 2x higher instance costs (approximately)
Performance Excellent, managed by AWS Same as default tenancy
Use Cases 99% of workloads Compliance requirements, BYOL licensing
Flexibility Can launch any instance type Limited instance type availability

âš ï¸ Important Considerations for Tenancy

Dedicated tenancy is expensive and rarely necessary. AWS's default tenancy provides robust isolation through hypervisor-level separation that meets most security and compliance requirements.

Only choose dedicated tenancy if you:

  • Have regulatory requirements mandating physical isolation
  • Have software licenses that require dedicated physical servers
  • Have organizational policies requiring physical separation

Note: Once you set a VPC to dedicated tenancy, you cannot change it back to default. Plan carefully before making this decision.

Section 2: Subnets & IP Addressing

Network Segmentation and IP Management

Understanding Subnets

A subnet is a logical subdivision of your VPC's IP address space. Subnets allow you to segment your network into smaller, more manageable pieces, providing organization, security, and fault tolerance. They're the building blocks of multi-tier application architectures in AWS.

ğŸ”‘ Critical Subnet Concept: Each subnet exists in exactly ONE Availability Zone and cannot span multiple AZs. This is a fundamental constraint that shapes how you design highly available architectures in AWS.

Key Subnet Characteristics:

  • Must be within the VPC's CIDR block range
  • Cannot overlap with other subnets in the same VPC
  • Bound to a single Availability Zone
  • Associated with exactly one route table
  • Can have a Network ACL for additional security

Public vs Private Subnets

Public Subnets

A public subnet has a route to an Internet Gateway (IGW) in its associated route table. Resources in public subnets can communicate directly with the internet in both directions (inbound and outbound).

Characteristics of Public Subnets:

  • Route table contains 0.0.0.0/0 â†’ Internet Gateway
  • Instances typically have public IP addresses or Elastic IPs
  • Direct bidirectional communication with the internet
  • Suitable for internet-facing resources

Common Resources in Public Subnets:

  • Web Servers: Nginx, Apache, IIS
  • Load Balancers: Application Load Balancers, Network Load Balancers
  • NAT Gateways: To provide internet access for private subnets
  • Bastion Hosts: Secure access points for SSH/RDP to private resources
  • Public APIs: API Gateway endpoints with custom domains

Private Subnets

A private subnet does NOT have a direct route to an Internet Gateway. Resources in private subnets cannot be accessed directly from the internet, providing an essential layer of security for sensitive workloads.

Characteristics of Private Subnets:

  • No direct route to Internet Gateway
  • Can route to NAT Gateway for outbound internet access
  • Instances only have private IP addresses
  • Not directly reachable from the internet

Common Resources in Private Subnets:

  • Application Servers: Business logic, microservices
  • Databases: RDS, Aurora, Redshift, DynamoDB VPC endpoints
  • Cache Layers: ElastiCache (Redis, Memcached)
  • Internal Services: Message queues, background workers
  • Sensitive Data Processing: ETL jobs, data analysis
